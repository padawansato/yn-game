## Context

yn-game は CEDEC 2008 で発表された「勇者のくせになまいきだ」の生態系シミュレーションを再現するプロジェクト。
現在、3種のモンスター（ニジリゴケ/ガジガジムシ/リザードマン）が移動・捕食・養分吸収/放出を行うが、
養分の保存則が保証されておらず、モンスターのライフサイクルは単調（生成→移動→死亡）に留まっている。

現在の養分漏出ポイント:
- `depleteOnDig()`: 掘削時に30%喪失（`NUTRIENT_DEPLETION_RATIO = 0.3`）
- `releaseNutrientsOnDeath()`: 隣接土壌がない場合に養分消失（L249: "Nutrients lost (entropy)"）
- `releaseNutrient()`: `MAX_NUTRIENT_PER_CELL` 超過時のクランプによる暗黙的消失

## Goals / Non-Goals

**Goals:**
- 養分保存則を確立し、全プロセスで `セルのnutrientAmount合計 + モンスターのcarryingNutrient合計 = 一定` を保証する
- つるはし操作でモンスターにダメージ/情報表示を可能にする
- 3種モンスターに原作準拠の変態・繁殖ライフサイクルを追加する
- 将来の魔分（`magicAmount`）追加に備えた拡張性を確保する

**Non-Goals:**
- 魔分システムの実装（将来の別changeで対応）
- 勇者AIの実装
- UIコンポーネントの詳細設計（コアロジックのみ）
- 原作の全モンスター亜種の実装（基本3種のライフサイクルのみ）

## Decisions

### D1: 養分保存則の実装方式

**決定:** 全セル（empty含む）に `nutrientAmount` を持たせ、全養分移動を明示的な転送として実装する。

**代替案:**
- A) 保存則をテストのみで保証 → 実行時に漏出を検知できず却下
- B) 養分をグローバルプールで管理 → セルごとの養分分布が失われ却下

**理由:** 空洞セルにも `nutrientAmount` を持たせることで、死亡時に隣接土壌がなくても養分を同セルに残せる。`getTotalNutrients()` は既に全セルを走査するため、保存則の検証コードの変更は最小限。

### D2: 養分放出範囲の拡大（4方向→9マス）

**決定:** 死亡時の養分放出を周囲8マス+中心の計9マスに均等分配する。放出先は土壌セルだけでなく空洞セルも対象とする。

**変更対象:**
- `getAdjacentSoilCells()` → `getSurroundingCells()` に改名・拡張（8方向+中心、soil/empty両対応）
- `releaseNutrientsOnDeath()` → 新しい放出範囲を使用

**理由:** 原作準拠。4方向のみだとモンスターが空洞の中央で死亡した場合に養分が消失しやすい。

### D3: `NUTRIENT_DEPLETION_RATIO` の廃止

**決定:** `NUTRIENT_DEPLETION_RATIO = 0.3` を削除し、`depleteOnDig()` を恒等関数（入力をそのまま返す）に変更するか、関数自体を削除する。

**理由:** 保存則に従い、掘削時の養分は100%モンスター生成/セルに保存される。

### D4: モンスターの変態段階（phase）の型設計

**決定:** `Monster` に `phase` フィールドを追加し、MonsterType ごとに異なる phase union 型を定義する。

```
NijirigokPhase = 'mobile' | 'bud' | 'flower' | 'withered'
GajigajimushiPhase = 'larva' | 'pupa' | 'adult'
LizardmanPhase = 'normal' | 'nesting' | 'laying' | 'egg'
```

**代替案:**
- A) phase を数値（0,1,2...）で管理 → 可読性が低く却下
- B) サブクラスで表現 → TypeScript のインターフェースベース設計に合わず却下
- C) 全モンスター共通の phase union → 無効な組み合わせ（ニジリゴケの'pupa'等）を型で防げず却下

**理由:** discriminated union により、各 MonsterType で有効な phase のみを許容し、コンパイル時に無効な遷移を検知できる。

### D5: 変態遷移の実行タイミング

**決定:** `tick()` の既存処理の後に `processPhaseTransitions()` を追加する。

tick の実行順序（変更後）:
1. 全モンスターの移動を計算（phase に応じて固定化判定）
2. コンフリクト解決
3. 移動適用
4. 捕食処理
5. 養分相互作用
6. 生命減少と死亡判定
7. **変態遷移チェック** ← 新規追加
8. **繁殖処理** ← 新規追加

**理由:** 変態は tick の最後に評価することで、その tick の移動・捕食・養分変化の結果を反映した判定ができる。

### D6: つるはしダメージの設計

**決定:** `attackMonster(state, monsterId, damage)` 関数を `simulation.ts` に追加する。

- 固定ダメージ値（初期値は定数で管理、調整可能）
- `digPower` は消費しない
- life が 0 以下になったら死亡処理（保存則に従った養分放出）
- 新イベント `MONSTER_ATTACKED` を追加

**理由:** 掘削と独立した操作として設計することで、将来の拡張（ダメージ種別、つるはし強化等）に対応しやすい。

### D7: 繁殖時の養分分配

**決定:** 繁殖時、親モンスターの `carryingNutrient` を子モンスター群に均等分配する。

- ニジリゴケ: 花が枯死時、carryingNutrient を子コケ（最大5匹）に均等分配
- ガジガジムシ: 成虫の carryingNutrient から一定量を消費して子（幼虫）を1匹生成
- リザードマン: 産卵時の養分消費、孵化時に親の養分の一部を子に移譲

**理由:** 保存則を遵守しつつ、繁殖による養分の再分配を実現する。

### D8: `MAX_NUTRIENT_PER_CELL` 超過時の扱い

**決定:** セルへの養分加算時、`MAX_NUTRIENT_PER_CELL` を超過する分は周囲のセルにオーバーフロー分配する。どこにも収まらない場合は元のセルで上限超過を許容する（保存則優先）。

**代替案:**
- A) 上限クランプ（現状） → 養分消失 = 保存則違反で却下
- B) 上限を撤廃 → バランス崩壊のリスクがあるが保存則は満たす

**理由:** 保存則を最優先とし、例外的にのみ上限超過を許容する。通常の養分循環ではオーバーフロー分配で十分対応できる。

## Risks / Trade-offs

**[パフォーマンス] 9マス探索の増加** → 4方向から9マスへの探索増加は O(1) の定数増加であり、問題にならない

**[複雑度] phase による条件分岐の増加** → 各システム（移動、養分、捕食）が phase を考慮する必要がある → 移動制御を movement/index.ts の `calculateMove()` 入口で phase チェックし、各パターン関数には影響を与えない設計で軽減

**[バランス] 養分喪失の廃止による養分過剰** → 掘削30%喪失がなくなることで養分が余る可能性 → 変態・繁殖による養分消費がカウンターバランスとなる。定数調整で対応

**[BREAKING] `depleteOnDig` 依存コードの破壊** → 既存テストの修正が必要 → テスト修正は tasks で管理

**[保存則の例外] `MAX_NUTRIENT_PER_CELL` 上限超過** → 保存則を優先し上限超過を許容するが、通常フローでは発生しにくい → オーバーフロー分配で対応

## Open Questions

- 各変態段階の具体的な遷移閾値（養分量、life値など）は実装時に調整
- つるはしの固定ダメージ値の初期設定
- 繁殖で生成される子モンスターの初期 life/carryingNutrient の具体値
- 花の攻撃（もよもよ）の射程・ダメージの詳細
