# テスト検証ルール

## 実装時の必須検証

1. **テスト実行**: 実装後は必ず `docker compose run --rm app pnpm test` で全テスト通過を確認
2. **lint**: `docker compose run --rm app pnpm lint` でエラーがないこと
3. **ブラウザ動作確認**: UI変更時は `docker compose up` でブラウザから実際にプレイして動作確認。スクリーンショットやコンソールログで検証する
4. **staleファイル注意**: `.js`, `.d.ts` がsrc/以下に残っているとViteが誤って読み込む。`find src -name "*.js" -o -name "*.d.ts"` で確認し、あれば削除

## TDD

- テストを先に書く（RED） → 最小実装（GREEN） → リファクタ
- Agent Teamsで並列実装する場合も、各チームメイトがTDDサイクルを守ること

## ライフサイクル統合テスト

ユニットテストだけでは仕様間の矛盾は発見できない。複数の仕様が連携するシステム（生態系、ゲームループ等）では、**端から端までの統合テスト**を書くこと。

例: 「苔がスポーンから繁殖まで自然に到達できるか」「虫がlarvaからadultまで成長できるか」

教訓: 移動コスト仕様（養分消費）とbud遷移条件（養分高+life低）が矛盾していたが、各ユニットテストは個別に通っていたため発見が遅れた。

## UI ロジックのテスト

Vue コンポーネント内の UI ロジックも **テスト対象として扱う**。`src/core/` の単体テストは充実していても、コンポーネント側のロジックがテスト未整備だと、リファクタや機能追加時のデグレ検出が既存 E2E 頼みになり、特定速度が落ちる。

### テスト方針

- **コンポーネントテスト**: `@vue/test-utils` + `vitest` で props → DOM / emit を検証
- **純粋関数の切り出し**: セル表示判定・クラス決定などの純粋関数は、可能なら別ファイルに切り出して単体テスト
- **characterization test**: 既存の描画ロジックをリファクタする前に、まず現状の DOM 出力を固めるテストを書く。構造を変えた後にそのテストが通れば振る舞いが保たれている証拠
- **モック最小**: コンポーネントテストは `gameState` を現実的な値で props に渡すだけで済む場合が多い。過剰なモックは避ける

### 500行ガイドライン

UI ロジックを単一コンポーネントに **500行以上** 埋め込まない。肥大化した時点で責務分離を検討する（CLAUDE.md「ファイル肥大化の閾値」を参照）。

### 教訓

`src/App.vue` の `getCellClass` / `getCellDisplay` などの UI ロジックはテスト完全未整備のまま、App.vue が 1149行まで肥大化した。結果、グリッド描画をリファクタする際の安全網が既存 E2E しかなく、機能追加のハードルが高くなっていた（2026-04 Issue #47）。
